# 问题定义与预期效果

## 这个库到底要解决什么问题
在工业 MAAS 迁移里，常见痛点是：
- 点位多（几百到几千），命名和语义不统一。
- 旧系统里目标函数、约束、安全边界分散在脚本和配置中。
- 新场景迁移靠人工拷贝改字典，容易漏字段或冲突。
- 发布前缺“可解释门禁”，导致上线风险高。

EasyShift-MaaS 的职责是把上面四件事标准化。

## 它不解决什么
- 它不是实时控制系统，不直接执行自动闭环控制。
- 它不是某个行业专属模型库，不绑定具体业务配方。

## 交付结果是什么
在这个库里，你最终交付的是：
1. `PointCatalog`: 现场点位到业务字段的统一映射。
2. `ScenarioTemplate`: 预测-优化逻辑的可版本化模板。
3. `MigrationDraft`: 模板迁移草案和风险说明。
4. `TemplateQualityReport`: 发布前质量结果。

## 预期效果怎么衡量
建议跟踪以下指标：
- 迁移效率：新场景落地周期（天/周）。
- 模板质量：质量门禁通过率、被阻断原因分布。
- 执行稳定性：仿真中安全规则触发率、不可解率。
- 维护成本：同类场景复用率。

## 为什么是“模板优先”
模板是可治理对象：
- 可版本化。
- 可审计。
- 可回归测试。
- 可比较新旧策略差异。

这比“把逻辑写在脚本里”更适合长期迭代与跨团队协作。
